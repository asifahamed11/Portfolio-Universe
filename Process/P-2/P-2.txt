Architectural Overhaul of the Portfolio Universe Extraction PipelineThe extraction of structured metadata from an extensive directory of developer portfolios presents a multi-dimensional data engineering challenge. The current operational pipeline for the "Portfolio Universe" project—relying on a single Kaggle T4 GPU, local instances of the Qwen2.5:7b model, and sequential fetching via the free Jina AI API—demonstrates a functional proof-of-concept but suffers from catastrophic bottlenecks. The primary failure points include Out-of-Memory (OOM) errors during batched inference due to strict Video Random Access Memory (VRAM) limits, idle timeouts inherent to unattended Kaggle sessions, and severe throughput degradation caused by third-party rate-limiting delays.To transform this system from a manual, multi-day ordeal into an autonomous, high-throughput, and entirely cost-free data pipeline, a fundamental paradigm shift is required. The solution demands migrating from local hardware constraints to a serverless orchestration model, transitioning from synchronous Application Programming Interface (API) scraping to localized asynchronous headless browsing, and upgrading from a locally hosted 7-billion parameter model to an enterprise-grade managed Large Language Model (LLM) offering a robust free tier.The following architectural analysis details a complete, step-by-step redesign that maximizes concurrency, eliminates strict external API rate limits, guarantees structured JavaScript Object Notation (JSON) output, and operates flawlessly in the background without incurring infrastructure costs.Transitioning the Execution Environment to Serverless OrchestrationThe reliance on Kaggle's interactive notebook environment introduces severe operational instability for long-running batch jobs. Kaggle notebooks are fundamentally designed for interactive data science and machine learning exploration, not continuous data pipeline execution. They routinely disconnect if left unattended, forcing manual interventions to restart interrupted processing loops. Furthermore, local inference on a single NVIDIA T4 GPU (which features 16GB of VRAM) restricts the processing to a batch size of one; any attempt to increase concurrency with a 7B parameter model leads directly to VRAM exhaustion and immediate process termination.Leveraging GitHub Actions as a Compute EngineThe optimal zero-cost alternative for long-running, scheduled workloads is GitHub Actions. While primarily recognized as a Continuous Integration and Continuous Deployment (CI/CD) platform, GitHub Actions provides a highly capable, ephemeral compute environment that is exceptionally well-suited for batch data processing. GitHub provides generous free tiers for public repositories, granting unlimited minutes on standard runners.A GitHub-hosted Ubuntu runner provides the ideal environment to execute asynchronous Node.js scripts autonomously. Crucially, GitHub Actions allows individual workflow jobs to run for up to six hours before encountering a hard termination limit. This six-hour window provides ample time for bulk data extraction when combined with concurrent processing strategies. Furthermore, the platform allows for advanced scheduling via standard cron syntax, or manual triggering via the workflow_dispatch event, meaning the system can execute entirely in the background without session timeouts or manual supervision.To understand the constraints of the GitHub Actions environment for scraping workloads, the following platform limits dictate the operational boundaries of the pipeline:Limit CategoryThresholdDescriptionJob Execution Time6 hoursThe maximum duration a single job can execute before failing.Workflow Run Time35 daysTotal allowed time including waiting for approvals (not applicable for automated scraping).API Request Limits1,000 per hourRestrictions on internal GitHub API calls (unrelated to external scraping).Storage Limits~1 GB soft limitThe repository size constraints, manageable via Git Large File Storage (LFS) if necessary.Concurrent JobsVariesLimit on parallel jobs executing across the repository simultaneously.Ephemeral State Management and Automated CommitsBecause GitHub Actions operates on ephemeral virtual machines, any data processed and extracted during the six-hour window will be permanently lost when the runner shuts down unless explicitly persisted. To ensure the 1,700+ processed portfolio Uniform Resource Locators (URLs) are permanently recorded, the workflow must programmatically commit the updated portfolios.json file back to the repository at the conclusion of the execution cycle.The git-auto-commit-action provides a seamless, battle-tested mechanism to detect changes to local files during a workflow run and push them to the main branch. To enable this functionality, the workflow must be explicitly configured with elevated repository permissions. By default, the GITHUB_TOKEN provided to actions has restricted read-only access to prevent malicious modifications. The workflow file must declare permissions: contents: write to grant the runner the authority to push the enriched JSON data back to the repository. This configuration entirely removes the need for external databases or third-party storage buckets, maintaining the 100% free constraint.Concurrency Controls and Idempotent DesignRunning scheduled workflows introduces the risk of overlapping executions, where a newly scheduled job begins before the previous job has completed its six-hour cycle. This race condition can lead to Git merge conflicts when multiple runners attempt to push modifications to portfolios.json simultaneously.GitHub Actions provides a native concurrency configuration block to manage this risk. By assigning a unique group identifier to the scraping job, the platform ensures that only one instance of the pipeline executes at any given time. For data extraction workflows where the goal is to process the entire queue without interruption, it is critical to configure cancel-in-progress: false. Setting this to true would prematurely terminate a running extraction job if a new trigger occurred, resulting in lost compute time and uncommitted data.Despite the six-hour runtime window, the architectural design must embrace idempotent, chunked processing. The Node.js script will read portfolios.json, parse the array, filter out URLs that already possess a valid metadata object, and process the next available subset of URLs. This idempotent approach ensures that if a workflow fails due to an unhandled exception, times out at the six-hour mark, or hits an external API quota, the subsequent execution will resume precisely where the previous execution halted, avoiding redundant API calls and duplicated effort.High-Fidelity, Zero-Cost Asynchronous Scraping ArchitectureThe current utilization of Jina AI's free extraction API forces a 10-second delay between requests to avoid HTTP 429 (Too Many Requests) blocks. When processing 1,700 URLs sequentially, this arbitrary delay alone accounts for over 4.7 hours of idle waiting time, completely separate from the actual network latency and LLM inference duration. Relying on external, rate-limited extraction endpoints fundamentally breaks the scalability of the pipeline.Evaluating Web Scraping API AlternativesThe market provides numerous specialized scraping APIs optimized for LLM ingestion, but they universally fail the strict zero-cost constraint at scale. Traditional HTTP clients (such as Axios in Node.js or Requests in Python) are fundamentally flawed for scraping modern web content, as they merely fetch the initial HTML payload delivered by the server. A vast majority of contemporary developer portfolios are built using modern JavaScript frameworks (React, Vue, Angular) deployed as Single Page Applications. When accessed via a standard HTTP client, these applications return an empty <div id="root"></div>, rendering the scrape useless because the actual content is generated client-side via JavaScript execution.Playwright automates Chromium, Firefox, and WebKit rendering engines directly via the Chrome DevTools Protocol (CDP), processing JavaScript-heavy applications exactly as a human user's browser would. By instructing Playwright to navigate to a portfolio URL and wait for specific lifecycle events—such as networkidle, which indicates that there have been no network connections for at least 500 milliseconds—the script guarantees that all asynchronous data fetches and dynamic DOM rendering have completed before extraction begins.Because Playwright is executing locally on the GitHub Actions runner, the only rate limit is the bandwidth of the runner itself and the specific rate-limiting policies of the individual target developer portfolios. Since these are distributed, independent websites hosted across various platforms (Vercel, Netlify, GitHub Pages), there is no centralized entity to enforce a global rate limit across the 1,700 URLs.Semantic DOM Pruning and Token OptimizationPassing a raw, fully computed HTML DOM directly to an LLM is a critical anti-pattern in data engineering. Raw HTML contains extensive tracking scripts, inline Cascading Style Sheets (CSS), scalable vector graphics (SVG) paths, base64 encoded images, and deep structural boilerplate (<div> nesting) that dilutes the semantic value of the text. This drastically inflates the token count, wastes API quotas, and increases the risk of LLM hallucinations or context window truncation.To solve this, the extraction pipeline must integrate two highly efficient, open-source libraries to perform a localized extraction transformation:Mozilla Readability.js: Originally designed to power Firefox's "Reader View," this standalone library evaluates DOM nodes based on heuristics such as text density, link-to-text ratios, and semantic HTML5 tags (e.g., <article>, <main>). It aggressively prunes navigation bars, footers, ad banners, and modal popups, isolating the primary readable content of the portfolio.Turndown: Following the isolation of the semantic DOM, the Turndown service converts the purified HTML into clean, well-structured Markdown.The conversion from raw, dynamic HTML to sanitized Markdown reduces LLM token consumption by up to 90% while preserving the essential semantic structure (headings, lists, links, and bold text) that language models require for accurate information extraction. This transformation significantly accelerates downstream LLM processing, ensures the payload fits comfortably within context windows, and strips away malicious or confusing markup that might distract the model's attention mechanism.Optimizing Playwright Execution in CI/CD EnvironmentsA known inefficiency of integrating Playwright into ephemeral CI/CD environments like GitHub Actions is the repeated downloading of browser binaries. Running npx playwright install --with-deps downloads between 300 to 500 Megabytes of browser executables (Chromium, Firefox, WebKit) and OS-level media codecs on every single workflow run. Over multiple runs, this installation phase wastes valuable minutes.To mitigate this, the workflow must leverage the actions/cache directive. By targeting the ~/.cache/ms-playwright directory and utilizing the repository's package-lock.json hash as the cache key, the runner will bypass the heavy download step on subsequent runs.YAML- name: Cache Playwright browsers
  uses: actions/cache@v4
  with:
    path: ~/.cache/ms-playwright
    key: ${{ runner.os }}-playwright-${{ hashFiles('package-lock.json') }}
This specific optimization saves approximately 30 to 60 seconds per CI execution, allowing the runner to dedicate more of its six-hour window to actual data extraction. Furthermore, to optimize memory on the runner, Playwright should be configured to only install Chromium, explicitly ignoring WebKit and Firefox, as cross-browser validation is irrelevant for a scraping workload.Managed LLM Integration for Structured ExtractionThe current use of Kaggle's T4 GPU to host the qwen2.5:7b model via Ollama introduces hard limits on batch processing due to VRAM constraints. Large Language Models demand significant memory just to load their parameter weights, leaving very little overhead for expanding context windows during inference. Attempting to process multiple portfolios concurrently on a single 16GB GPU inevitably results in an Out-Of-Memory failure. By shifting the LLM inference from local hardware to a managed API provider with a generous free tier, the computational burden is offloaded entirely to cloud infrastructure, unlocking the potential for massive concurrency.Navigating the Multi-Model Gateway LandscapeIn the modern AI ecosystem, developers frequently utilize LLM gateways to abstract the complexities of provider-specific APIs, manage fallbacks, and track token usage. Tools like LiteLLM and OpenRouter dominate this space. LiteLLM acts as a universal proxy, standardizing over 100 providers into an OpenAI-compatible format, and is highly recommended for self-hosted teams seeking robust fallback mechanisms and load balancing. OpenRouter offers a managed Software-as-a-Service (SaaS) equivalent, providing instant access to a vast array of open-source and proprietary models through a unified endpoint.While OpenRouter does offer access to several free models (such as Meta's Llama variants or Mistral), its free tier is explicitly designed for experimentation rather than production volume. OpenRouter enforces a strict limit of 20 Requests Per Minute (RPM) and a daily cap of 50 requests (extendable to 1,000 only if a minimal account balance is maintained). Given the dataset size of 1,700 URLs, relying on OpenRouter's unverified free tiers risks unpredictable rate limiting and service interruptions. Therefore, integrating directly with a primary AI provider offering a formalized developer free tier is the superior architectural choice.Free Tier Benchmarking and Provider SelectionA comprehensive analysis of current free-tier LLM API providers reveals the precise constraints that dictate pipeline design. The generous token allowance entirely mitigates the constraints found in providers like Groq.Enforcing Strict JSON Outputs via Response SchemasA common point of failure in LLM data pipelines is the reliance on traditional prompt engineering to extract structured data. Instructing a model to "output only valid JSON" is notoriously fragile; models frequently prepend conversational filler (e.g., "Here is the JSON you requested:") or fail to escape characters properly, resulting in catastrophic parsing errors in the Node.js application.The Gemini API natively resolves this through its Structured Outputs feature, controlled via the responseSchema configuration parameter. By passing a predefined OpenAPI 3.0 schema object to the generateContent method and explicitly setting the responseMimeType to application/json, the API enforces strict adherence to the schema at the generation level.When the schema defines specific properties (Name, Role, Tech Stack, 50-word Summary, SEO Score, Hire Status) and assigns data types (String, Integer, Array), the Gemini model alters its internal decoding process to guarantee that the output conforms identically to the required structure. If a field is specified as an enum (e.g., forcing Hire Status to only equal "Open", "Closed", or "Unknown"), the model is mathematically constrained from generating any out-of-bounds vocabulary. This capability entirely eliminates the need for complex regular expressions, string cleaning, or error-prone try-catch JSON parsing logic within the scraping script.Advanced Concurrency Engineering and Queue ManagementMoving away from the synchronous, single-threaded execution loop unlocks the ability to process URLs asynchronously. However, launching thousands of simultaneous Playwright instances and Gemini API calls via an unconstrained Promise.all() structure will immediately crash the GitHub Actions runner due to memory exhaustion and trigger instantaneous IP bans from the Gemini API.To extract maximum performance while strictly adhering to Google's 15 RPM limit, the pipeline requires an advanced, temporal concurrency controller.Designing the Throttling MechanismIn the Node.js ecosystem, developers often turn to basic concurrency libraries like p-limit or async. While p-limit excels at establishing a maximum ceiling for simultaneous active promises (e.g., ensuring only 10 tasks run in parallel), it lacks any awareness of time. If 15 tasks complete in 10 seconds, p-limit will immediately launch the next 15 tasks, vastly exceeding a provider's requests-per-minute restriction.The p-queue library is the definitive engineering solution for this architecture. It operates as a robust priority queue equipped with sliding-window rate-limiting capabilities.By configuring p-queue with highly specific temporal parameters, the Node.js execution loop is throttled precisely to the API's constraints without wasting idle time:Memory Protection: Setting concurrency: 5 ensures that no more than five Playwright browser tabs are instantiated and rendering DOMs simultaneously. This prevents the CI runner from thrashing or crashing due to browser memory bloat.API Compliance: Setting intervalCap: 15 and interval: 60000 enforces that exactly 15 requests are dispatched every 60,000 milliseconds (one minute). This perfectly mirrors the Gemini free tier limit.Burst Prevention: Activating strict: true enables a sliding window algorithm rather than a fixed window. In a fixed window, 15 tasks might execute at second 59, and another 15 at second 61 of the next minute, causing a massive 30-request burst that triggers a 429 block. The sliding window guarantees a smooth, evenly distributed execution rate that never violates the rolling limit.Resiliency Through Exponential Backoff and JitterEven with mathematically perfect queueing, distributed systems are inherently unstable. Network connectivity drops, target portfolios experience DNS timeouts, anti-bot mechanisms trigger falsely, or the LLM provider returns temporary 500 Internal Server Error codes during peak usage periods. A robust data engineering pipeline must anticipate, absorb, and recover from these faults without halting the entire workflow.Wrapping the core fetch-and-extract logic in a custom retry handler ensures systemic resilience. If a specific developer portfolio triggers a Playwright navigation timeout, the script catches the exception, logs the failure, and schedules a retry.Crucially, these retries must implement an exponential backoff algorithm combined with randomization (jitter). If an API endpoint is struggling, hammering it with immediate retries exacerbates the failure. Instead, the script pauses for a delay that doubles with each attempt (e.g., 2 seconds, 4 seconds, 8 seconds), adding a random jitter of a few hundred milliseconds to prevent synchronized retry storms from multiple concurrent workers. If the maximum retry threshold (e.g., three attempts) is exhausted, the URL is logged as an unrecoverable failure, and the queue proceeds to the next portfolio, ensuring the pipeline never stalls indefinitely.Comprehensive Systems Architecture BlueprintThe synthesis of these advanced technologies yields a robust, highly parallelized, zero-cost data extraction pipeline. 2. Node.js Conceptual Code Architecture (src/pipeline.js)The core Node.js application acts as the conductor. It initializes the headless browser, isolates unprocessed URLs, manages the strict rate-limiting queue, sanitizes the volatile DOM into stable Markdown, and enforces the OpenAPI schema on the Gemini model.JavaScriptimport fs from 'fs/promises';
import { chromium } from 'playwright';
import { Readability } from '@mozilla/readability';
import TurndownService from 'turndown';
import { GoogleGenerativeAI } from '@google/generative-ai';
import PQueue from 'p-queue';
import { JSDOM } from 'jsdom';

// Initialize Gemini SDK with securely injected credentials
const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);

// Define strict OpenAPI 3.0 schema to guarantee structured JSON output
const portfolioSchema = {
  type: "OBJECT",
  properties: {
    Name: { type: "STRING", description: "The developer's full name" },
    Role: { type: "STRING", description: "Primary job title or role" },
    TechStack: { 
      type: "ARRAY", 
      items: { type: "STRING" }, 
      description: "Comprehensive list of technologies, frameworks, and languages" 
    },
    Summary: { type: "STRING", description: "A concise 50-word summary of the developer's background" },
    SEOScore: { type: "INTEGER", description: "Estimated SEO quality score (1-100) based on content structure" },
    HireStatus: { 
      type: "STRING", 
      enum: ["Open", "Not available", "Unknown"],
      description: "Availability for hire" 
    }
  },
  required: ["Name", "Role", "TechStack", "Summary", "SEOScore", "HireStatus"]
};

// Initialize the LLM with the Flash model and enforce the schema configuration
const model = genAI.getGenerativeModel({
  model: "gemini-2.5-flash",
  generationConfig: {
    responseMimeType: "application/json",
    responseSchema: portfolioSchema,
  }
});

// Configure Advanced Rate Limiting Queue
// Enforces Google's strict 15 Requests Per Minute limit using a sliding window
const queue = new PQueue({
  concurrency: 5, // Prevent CI runner memory bloat by limiting active tabs
  intervalCap: 15, // Maximum 15 actions
  interval: 60000, // Per 60 seconds
  strict: true     // Enable sliding window to prevent API burst blocking
});

async function extractMarkdown(url, browser) {
  const context = await browser.newContext();
  const page = await context.newPage();
  
  try {
    // Navigate with a timeout, wait for network activity to settle to capture SPAs
    await page.goto(url, { waitUntil: 'networkidle', timeout: 45000 });
    
    // Extract the fully rendered HTML DOM
    const html = await page.content();
    
    // Prune DOM and strip boilerplate (navs, footers) using Readability
    const dom = new JSDOM(html, { url });
    const reader = new Readability(dom.window.document);
    const article = reader.parse();
    
    if (!article || !article.content) throw new Error("Unreadable content or anti-bot block");

    // Convert refined semantic HTML to Markdown to drastically reduce LLM tokens
    const turndownService = new TurndownService();
    return turndownService.turndown(article.content);
  } finally {
    // Ensure memory is freed regardless of execution success
    await context.close();
  }
}

async function processUrl(url, browser, retries = 3) {
  try {
    const markdownContext = await extractMarkdown(url, browser);
    
    const prompt = `
      Analyze the following markdown content extracted from a developer's portfolio website.
      Extract the specific data points required by the JSON schema.
      If a data point is entirely missing, infer it reasonably from context or return 'Unknown'.
      
      PORTFOLIO CONTENT:
      ${markdownContext}
    `;

    // Await structured generation from Gemini
    const result = await model.generateContent(prompt);
    
    // Output is guaranteed to be valid JSON string due to responseMimeType configuration
    return JSON.parse(result.response.text());

  } catch (error) {
    if (retries > 0) {
      // Exponential backoff with jitter to recover from transient faults
      const backoffDelay = (1000 * Math.pow(2, 4 - retries)) + Math.floor(Math.random() * 500);
      console.warn(`[Retry] Fault on ${url}. Retrying in ${backoffDelay}ms...`);
      await new Promise(res => setTimeout(res, backoffDelay));
      return processUrl(url, browser, retries - 1);
    } else {
      console.error(`[Error] Unrecoverable failure processing ${url}. Skipping.`);
      return null; 
    }
  }
}

async function runPipeline() {
  // Launch Playwright in headless mode
  const browser = await chromium.launch({ headless: true });
  
  // Load current database state
  const data = JSON.parse(await fs.readFile('portfolios.json', 'utf8'));
  
  // Isolate URLs that lack the extractedMetadata object
  const pendingUrls = data.filter(entry => !entry.extractedMetadata);
  
  // Process up to 1,490 URLs (Safely under Google's 1,500 daily free tier limit)
  const chunkToProcess = pendingUrls.slice(0, 1490); 

  console.log(`Initiating asynchronous extraction for ${chunkToProcess.length} portfolios...`);

  // Map tasks into the strict rate-limiting queue
  const processingPromises = chunkToProcess.map(entry => 
    queue.add(async () => {
      console.log(`Processing URL: ${entry.url}`);
      const metadata = await processUrl(entry.url, browser);
      
      if (metadata) {
        entry.extractedMetadata = metadata; // Mutate local object with structured JSON
      }
    })
  );

  // Await the completion of the entire queued batch
  await Promise.all(processingPromises);
  
  // Persist the enriched state back to disk for the GitHub Action to commit
  await fs.writeFile('portfolios.json', JSON.stringify(data, null, 2));
  console.log("Pipeline execution complete. Database state updated.");
  
  await browser.close();
}

// Execute and catch unhandled promise rejections
runPipeline().catch(console.error);
System Outcomes and Strategic AdvantagesThe comprehensive implementation of this redesigned architecture directly confronts and resolves the inherent limitations of the legacy Kaggle-centric setup, providing a deeply resilient data engineering framework.First, the transition to GitHub Actions completely eliminates the necessity for manual supervision. Because workflows execute natively in the cloud and automatically terminate upon completion, interactive session timeouts are neutralized. Additionally, utilizing Gemini's native responseSchema OpenAPI configuration eradicates the parsing inconsistencies that plague standard prompt engineering, ensuring that every successful extraction maps cleanly and reliably into the required schema.Finally, the introduction of p-queue demonstrates advanced rate limit engineering. By converting a disorganized flood of parallel requests into a strictly paced, sliding-window stream (15 requests per minute), the architecture acts in perfect harmony with Google's free-tier restrictions. Combined with exponential backoff and randomized jitter to absorb transient network faults, this prevents temporary IP bans or API lockouts, ensuring continuous, predictable throughput.This comprehensive architectural overhaul transforms the extraction process into an enterprise-ready pipeline capable of processing thousands of dynamically rendered web pages concurrently, precisely, and entirely free of charge.
